ZeroLeakX scenario simulator
Each scenario feeds synthetic behavioural input through the real ML model on the server — the detection, scoring and SHAP explanation are genuine. Watch the SOC panel (right) react live.
Scenario 1 — Credential-stuffing takeover
Attacker logs in with stolen credentials from a new device + location and types with a different rhythm. The model scores keystroke, device and geo all diverging over ~8s. DTS crosses critical → step-up OTP → attacker fails → session killed.
Scenario 2 — Mid-session behavioural takeover
Same device, but a different person takes over the keyboard mid-session. Only keystroke + pointer dynamics drift. DTS dips to high-risk → one-time step-up challenge, no hard block. Pure behavioural-biometric detection.
Scenario 3 — Geo anomaly (travel)
Legitimate user logs in from a city 380 km away. Geo context collapses, but it is a low-weight contextual signal by design — so with clean keystroke + device behaviour the DTS only dips slightly. The event is logged and monitoring tightens, but the user is NOT challenged. Correct: travel alone shouldn't punish a genuine user.
Scenario 4 — New device login
User logs in from an unrecognised device. Device fingerprint drops to 0; behaviour stays consistent. DTS falls to high — step-up OTP triggered, but session resumes after verification (device added to trusted list).
Scenario 5 — Legitimate session
Type naturally in the transfer form and watch keystroke dynamics respond to your real typing. DTS stays high, no alerts, transfer proceeds seamlessly.
